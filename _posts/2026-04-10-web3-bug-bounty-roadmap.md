## Web3 버그 바운티 플랫폼

### Immunefi — 1순위

Web3 버그 바운티의 **절대 왕자**.

- Web3 전문 플랫폼
- Critical 보상: **$10,000 ~ $10,000,000**
- 총 지급 보상: **$1억 이상**
- 대부분의 주요 DeFi 프로토콜이 여기에 프로그램 운영

**시작 방법:**
1. 프로그램 목록에서 관심 있는 프로젝트 선택
2. Scope(범위) 확인 — 어떤 컨트랙트가 대상인지
3. 코드 읽기
4. 취약점 발견 시 보고서 작성
5. 트리아지 → 검증 → 보상

### Code4rena — 경쟁형 감사

- **감사 대회** 방식. 정해진 기간 안에 여러 사람이 취약점을 찾음
- 기간: 보통 3~7일
- 상금 풀: $20,000 ~ $200,000+
- **초보자에게 좋다** — 다른 사람의 보고서를 보고 배울 수 있음
- 대회 끝나면 **모든 발견 사항이 공개**됨

### Sherlock

- Code4rena와 비슷한 경쟁형 감사
- 고정 보상 + 순위 기반 보상
- 감사 품질에 대한 **보험** 제공

### Hats Finance

- 탈중앙화 버그 바운티 프로토콜
- 온체인에서 보상 관리

## 보상 등급

| 등급 | 설명 | 일반적 보상 |
|------|------|------------|
| **Critical** | 자금 직접 탈취 가능 | $50,000 ~ $10,000,000 |
| **High** | 자금 손실 가능성 높음 | $10,000 ~ $50,000 |
| **Medium** | 제한적 영향 | $1,000 ~ $10,000 |
| **Low** | 낮은 영향, 비효율 등 | $100 ~ $1,000 |

Critical 하나면 **연봉급**이 나올 수 있다.

### Phase 2: 실전 감각 키우기 (2~4개월)

```
□ Code4rena 과거 대회 보고서 읽기 (매일 1~2개)
```

**과거 보고서 읽기가 가장 효율적인 학습법이다.** 실제로 돈이 걸렸던 취약점이 어떻게 발견됐는지 보는 것.

### Phase 3: 독립 헌터 (4개월~)

```
□ Immunefi에서 프로그램 선택 후 본격 헌팅
□ 자체 보안 도구 세팅 (Slither, Foundry 테스트)
□ 특정 분야 전문화 (렌딩, DEX, 브릿지 등)
□ 커뮤니티 참여 (Discord, Twitter)
□ 첫 번째 유효 보고서 제출
```

## 보고서 작성법

좋은 보고서의 구조:

```markdown
## 제목
간결하게 취약점을 설명

## 심각도
Critical / High / Medium / Low

## 요약
1~2문장으로 핵심 설명

## 취약점 상세
- 어떤 컨트랙트의 어떤 함수가 문제인지
- 왜 문제인지 (코드 레벨)
- 전제 조건이 있다면 명시

## 공격 시나리오
1. 공격자가 X를 한다
2. Y가 발생한다
3. 결과: Z만큼의 자금 탈취

## PoC (Proof of Concept)
Foundry 테스트 코드로 실제 익스플로잇 증명

## 영향
최대 피해 금액, 영향 범위

## 수정 권장사항
구체적인 코드 수정 방안
```

**PoC가 없으면 보상 확률이 급감한다.** Foundry로 테스트 코드를 작성하는 능력이 필수.

## 마인드셋

### 1. 프로토콜을 이해하라

코드를 바로 읽지 말고, **문서(docs)를 먼저 읽어라.** 프로토콜이 뭘 하려는 건지 이해해야 "이게 의도된 동작인가, 버그인가"를 판단할 수 있다.

### 2. 경계값을 생각하라

- 잔고가 0이면? 최대값이면?
- 첫 번째 사용자면? 마지막 사용자면?
- 풀에 유동성이 없으면?
- 토큰 decimals가 6이면? 18이면?

### 3. 조합을 생각하라

단일 함수는 안전해도, **여러 함수를 특정 순서로 호출하면** 취약점이 생기는 경우가 많다.

### 4. 포기하지 마라

첫 유효 보고서까지 **수개월**이 걸릴 수 있다. 정상이다.

> "모든 감사자의 첫 번째 찾은 버그 이전에는 수백 시간의 빈손이 있었다."
