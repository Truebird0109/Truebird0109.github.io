## 토큰이 뭔데?

**ETH**는 이더리움 블록체인의 **기본 화폐**(네이티브 토큰)다.

근데 이더리움 위에 **자기만의 화폐**를 만들 수 있다. 이게 **토큰**이다.

> USDT, USDC, UNI, LINK, SHIB... 다 이더리움 위의 토큰이다.

토큰은 실체가 있는 게 아니라, **스마트 컨트랙트의 숫자 기록**이다.

```
컨트랙트 내부:
balances[0xA...] = 1000   ← A가 토큰 1000개 보유
balances[0xB...] = 500    ← B가 토큰 500개 보유
```

---

## 왜 표준이 필요한가?

모든 사람이 제각각 토큰을 만들면?
- 거래소: "이 토큰 연동하려면 새로 개발해야 하네..."
- 지갑: "이 토큰은 transfer(), 저건 send(), 또 다른건 move()..."
- DeFi: "호환 불가"

그래서 **"토큰은 이 함수들을 반드시 가지고 있어야 한다"**는 약속을 만들었다.
이게 **ERC-20** 표준이다.

---

## ERC-20 필수 함수

```solidity
interface IERC20 {
    // 조회
    function totalSupply() external view returns (uint256);
    function balanceOf(address account) external view returns (uint256);
    
    // 전송
    function transfer(address to, uint256 amount) external returns (bool);
    
    // 위임 전송
    function allowance(address owner, address spender) external view returns (uint256);
    function approve(address spender, uint256 amount) external returns (bool);
    function transferFrom(address from, address to, uint256 amount) external returns (bool);
    
    // 이벤트
    event Transfer(address indexed from, address indexed to, uint256 value);
    event Approval(address indexed owner, address indexed spender, uint256 value);
}
```

### 핵심 흐름 2가지

**직접 전송:**
```
A → transfer(B, 100) → B에게 100 토큰 전송
```

**위임 전송 (approve + transferFrom):**
```
1. A → approve(DEX, 1000)     // "DEX야, 내 토큰 1000개까지 써도 돼"
2. DEX → transferFrom(A, C, 500)  // DEX가 A의 토큰 500개를 C에게 보냄
```

DeFi에서는 거의 항상 **위임 전송**을 쓴다. 스왑, 스테이킹, 렌딩 전부.

## approve의 함정 — 보안 핵심

### 무한 승인(Infinite Approval)

```solidity
// 많은 DApp이 이렇게 요청한다
token.approve(dex_address, type(uint256).max);
// = 2^256 - 1 ≈ 사실상 무한
```

**편하다.** 매번 approve 안 해도 되니까.

**위험하다.** 그 DApp이 해킹당하면? → 승인한 토큰 **전부 탈취 가능**.

실제 사건들:
- 2022년 BadgerDAO 해킹: 프론트엔드 해킹 → 악의적 approve 유도 → **1.2억 달러 탈취**
- 수많은 피싱 사이트: "이 NFT 민팅하세요" → 실은 approve(해커, 무한) 서명 유도

### 방어법

1. **필요한 만큼만 approve** (정확한 금액)
2. **사용 후 approve를 0으로 리셋**
3. **revoke.cash** 같은 도구로 정기적으로 승인 현황 점검

## 간단한 ERC-20 구현

```solidity
contract SimpleToken is IERC20 {
    string public name = "SimpleToken";
    string public symbol = "SIM";
    uint8 public decimals = 18;
    uint256 public totalSupply;
    
    mapping(address => uint256) public balanceOf;
    mapping(address => mapping(address => uint256)) public allowance;
    
    constructor(uint256 _supply) {
        totalSupply = _supply;
        balanceOf[msg.sender] = _supply;
    }
    
    function transfer(address to, uint256 amount) external returns (bool) {
        require(balanceOf[msg.sender] >= amount, "Insufficient");
        balanceOf[msg.sender] -= amount;
        balanceOf[to] += amount;
        emit Transfer(msg.sender, to, amount);
        return true;
    }
    
    function approve(address spender, uint256 amount) external returns (bool) {
        allowance[msg.sender][spender] = amount;
        emit Approval(msg.sender, spender, amount);
        return true;
    }
    
    function transferFrom(address from, address to, uint256 amount) external returns (bool) {
        require(balanceOf[from] >= amount, "Insufficient");
        require(allowance[from][msg.sender] >= amount, "Not approved");
        
        balanceOf[from] -= amount;
        balanceOf[to] += amount;
        allowance[from][msg.sender] -= amount;
        
        emit Transfer(from, to, amount);
        return true;
    }
}
```

## decimals — 소수점의 비밀

블록체인에는 소수점이 없다. 정수만 다룬다.

```
decimals = 18이면:
1 토큰 = 1,000,000,000,000,000,000 (10^18)

내가 "1.5 토큰" 보내고 싶으면:
실제 전송값 = 1,500,000,000,000,000,000
```

**보안 이슈**: decimals를 잘못 처리하면 **금액이 틀어진다.** 6 decimals(USDC) 토큰에 18 decimals 계산을 적용하면 난리남.

## 정리

- 토큰 = 스마트 컨트랙트의 잔고 기록
- ERC-20 = 토큰의 공통 규약 (transfer, approve, transferFrom)
- approve는 편하지만 **무한 승인은 위험**
- decimals 처리 실수 = 금액 오류 취약점
- approve 피싱은 Web3에서 **가장 흔한 공격** 중 하나
