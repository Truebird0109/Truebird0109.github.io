## 왜 스마트 컨트랙트가 해킹당하는가?

일반 웹사이트와 근본적으로 다른 특성이 있습니다.

| | 웹사이트 | 스마트 컨트랙트 |
|---|---|---|
| 코드 공개 여부 | 서버 코드 비공개 | **모든 코드 공개** |
| 수정 가능성 | 패치 배포로 즉시 수정 가능 | **수정 불가** |
| 보관 자산 | DB의 숫자 기록 | **실제 자금이 컨트랙트 내부에 존재** |
| 실행 환경 | 운영자의 서버 | **전 세계 공개 네트워크** |

코드가 공개되어 있고, 내부에 실제 자금이 있으며, 수정할 수 없습니다.

**공격자 입장에서 이보다 좋은 환경이 없습니다.**

---

## 공격 표면 맵

```
                    ┌─────────────────────┐
                    │   Frontend (Web)     │ ← 피싱, 악성 approve 유도
                    └──────────┬──────────┘
                               │
                    ┌──────────▼──────────┐
                    │   RPC / Node        │ ← MEV, Front-running
                    └──────────┬──────────┘
                               │
┌──────────────┐   ┌──────────▼──────────┐   ┌──────────────┐
│ External     │◄──│  Smart Contract     │──►│ Oracle       │
│ Contracts    │   │  (주요 공격 대상)    │   │ (가격 데이터) │
└──────────────┘   └──────────┬──────────┘   └──────────────┘
                               │
                    ┌──────────▼──────────┐
                    │   Blockchain State   │ ← 상태 조작
                    └─────────────────────┘
```

공격 벡터가 사방에 존재합니다. 컨트랙트 자체의 취약점, 외부 의존성, 프론트엔드, 인프라 모두가 공격 대상입니다.

## 대표 취약점 카테고리

### Tier 1: 자주 발생하고 피해가 큰 취약점

| 취약점 | 한 줄 설명 | 다루는 편 |
|--------|-----------|-----------|
| **Reentrancy** | 함수 실행 중에 재진입하여 자금 탈취 | #11 |
| **Access Control** | 관리자 전용 함수를 일반 사용자가 호출 | #12 |
| **Flash Loan Attack** | 무담보 대출로 시장 조작 | #13 |
| **Oracle Manipulation** | 가격 데이터 조작 | #14 |

### Tier 2: 반드시 알아야 할 취약점

| 취약점 | 한 줄 설명 |
|--------|-----------|
| **Integer Overflow/Underflow** | 숫자가 범위를 초과해 잘못된 값 (0.8.0 이후 자동 방지) |
| **Front-running / MEV** | 내 트랜잭션을 미리 확인하고 앞에 끼어드는 공격 |
| **Denial of Service** | 컨트랙트 실행을 불가능하게 만드는 공격 |
| **Signature Replay** | 이미 사용한 서명을 재사용하는 공격 |
| **Delegatecall Injection** | 외부 코드를 내 컨텍스트에서 실행하는 공격 |

## 보안 감사(Audit)란?

스마트 컨트랙트를 배포하기 전에 **전문가가 코드를 리뷰**하는 절차입니다.

```
프로젝트 → 감사 업체에 코드 제출
  ↓
감사자: 코드를 라인 단위로 검토하며 취약점 탐색 (1~4주)
  ↓
감사 보고서 발행 (Critical / High / Medium / Low / Info)
  ↓
프로젝트: 취약점 수정
  ↓
배포
```

대표적인 감사 업체:
- **Trail of Bits** — 최고 수준. 해킹 대회 우승 출신들로 구성
- **OpenZeppelin** — 표준 라이브러리도 개발하는 감사 업체
- **Consensys Diligence** — 이더리움 생태계 핵심 업체
- **Spearbit** — 독립 감사자 네트워크

감사 비용은 소규모 프로젝트도 **수천만 원**, 대형 프로토콜은 **수억 원**에 달합니다.

## 주요 보안 도구

### 정적 분석 (코드 분석으로 취약점 탐지)

- **Slither** — Trail of Bits 개발. 가장 널리 사용되는 도구
- **Mythril** — EVM 바이트코드 분석
- **Semgrep** — 패턴 매칭 기반 (Web2에서도 유명)

### 퍼징 (무작위 입력으로 버그 탐지)

- **Echidna** — 솔리디티 전용 퍼저
- **Foundry (forge fuzz)** — 현재 가장 널리 사용되는 도구

### 형식 검증 (수학적으로 안전성 증명)

- **Certora** — "이 함수는 절대 이 상태가 될 수 없다"를 수학적으로 증명
- **Halmos** — 심볼릭 실행 기반

## 보안 마인드셋

코드를 읽을 때 항상 다음 질문을 던지는 습관을 만드세요:

1. **"이 함수를 아무나 호출하면 어떻게 됩니까?"** → Access Control
2. **"외부 호출 이후에 상태를 변경하면?"** → Reentrancy
3. **"이 숫자가 0이면? 최대값이면?"** → Edge Case
4. **"이 가격 데이터는 어디서 가져옵니까?"** → Oracle 의존성
5. **"한 트랜잭션 안에서 이것을 반복하면?"** → Flash Loan 가능성

이 5가지 질문을 습관화하면 **상당수의 취약점**을 발견할 수 있습니다.

---

## 정리

- 스마트 컨트랙트: 코드 공개 + 자금 보유 + 수정 불가 = 공격자에게 매우 유리한 환경
- 대표 취약점: Reentrancy, Access Control, Flash Loan, Oracle Manipulation
- 보안 감사 = 배포 전 전문가 리뷰 (비용이 높습니다)
- Slither, Echidna, Foundry 등 도구 활용
- **5가지 질문**을 습관화하세요
